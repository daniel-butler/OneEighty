MetronomeEngine - Key Methods

L97-111togglePlayback()
Flips isPlaying. Calls start/stopMetronome(). Updates SharedMetronomeState, LiveActivityManager, NowPlaying, onStateChange callback.
L113-131incrementBPM() / decrementBPM()
+/- 1 BPM (clamped 150-230). If playing, triggers handleBPMChange() which debounces 0.3s before restarting timer.
L53-72setup()
Full init: resets to defaults (180 BPM, stopped), sets up audio engine, remote commands, state observer, interruption observer, Now Playing.
L76-85ensureReady()
Lightweight init for background wake. Same as setup() but preserves current state. No-ops if already set up.
L263-277handleBPMChange()
Stops timer immediately. Waits 0.3s debounce. Restarts timer with new BPM. Prevents rapid tick restarts during +/- button mashing.
L387-437setupRemoteCommands() / updateNowPlaying()
Registers play/pause/toggle + next/prev track (mapped to BPM +/-) with MPRemoteCommandCenter. Updates MPNowPlayingInfoCenter with "BPM SPM" title.

PhoneSessionManager - Key Methods

L103-131handleWatchCommand()
Dispatches: "toggle", "start", "stop", "incrementBPM", "decrementBPM". Calls engine.ensureReady() first for background wake. Sends updated state back to watch.
L35-60sendStateToWatch()
Always calls updateApplicationContext() (persists). Also sendMessage() if reachable (immediate). Payload: {bpm, isPlaying}.

LiveActivityManager - Key Methods

L63-86startActivity()
Creates new Live Activity via Activity.request(). Ends any existing one first. Starts observing contentUpdates for delivery confirmation.
L88-139updateActivity()
Critical updates (play/stop): bypass throttle, push immediately. Normal updates (BPM): coalesced via timer. If no current activity exists, falls back to startActivity() (L95-102).

AudioSessionManager

L21-29activate()
Sets AVAudioSession to .playback + .mixWithOthers. This allows ticks to play alongside audiobooks/music without interrupting them.
L48-69handleInterruption()
On .began: posts .audioInterruptionBegan. On .ended with .shouldResume: posts .audioInterruptionEnded. MetronomeEngine observes these.

AppIntents - Cross-Process Communication

L48-61ToggleMetronomeIntent
Reads isPlaying from SharedMetronomeState, flips it, writes back, pushes activity update, posts Darwin command (start or stop).
L101-131Increment/DecrementBPMIntent
Reads BPM from SharedMetronomeState, +/- 1 (clamped 150-230), writes back, pushes activity update, posts Darwin stateChanged.
Key insight: Intents run in the widget extension process, NOT the app process. They communicate with the app via SharedMetronomeState (UserDefaults) and Darwin notifications. The app's StateChangeObserver picks up these notifications and drives the engine.

isPlaying: Bool default: false
UserDefaults key "isPlaying". Used by ToggleMetronomeIntent to decide start vs stop.
volume: Float default: 0.4
UserDefaults key "volume".
App Group: group.com.danielbutler.MetronomeApp
Darwin Notifications: stateChanged, command.start, command.stop

WatchSessionManager - Key Methods

L53-78sendCommand()
If reachable: sendMessage with reply handler (gets state back). On error: falls back to transferUserInfo. If unreachable: transferUserInfo (queued delivery).

Handled Audio interrupted by another app (Audible, Music)

▶

When another app takes over the audio session (e.g., starting an audiobook), iOS posts an interruption notification.

iOS
AVAudioSession.interruptionNotification fires with .began
AudioSessionManager
Posts .audioInterruptionBegan

MetronomeEngine
Saves wasPlayingBeforeInterruption, stops metronome, sets isPlaying=false, updates Live Activity + Now Playing
MetronomeEngine.swift:356-368
iOS
When interruption ends with .shouldResume, engine re-activates audio session and resumes playback
MetronomeEngine.swift:370-383

With .mixWithOthers: This interruption flow is less likely since we mix alongside other audio. It mainly triggers for phone calls or Siri.

Handled BPM change while playing (debounce)

▶

Rapidly tapping +/- while the metronome is playing would cause jarring tick restarts. The engine debounces.

MetronomeEngine
incrementBPM() called → stopMetronome() immediately (no more ticks)
MetronomeEngine.swift:266
MetronomeEngine
Schedules timer for 0.3s. If another +/- happens before timer fires, timer resets.
MetronomeEngine.swift:267-276
MetronomeEngine
After 0.3s of no changes: startMetronome() with new BPM
MetronomeEngine.swift:273-274

UX effect: Ticks pause briefly during rapid BPM adjustment, then resume at the new tempo. LiveActivity + NowPlaying update on each tap (no debounce on those).

Limitation Starting Live Activity from background (watch-initiated)

▶

iOS requires the app to be in the foreground to start a new Live Activity via Activity.request(). When the watch starts playback, the phone may be backgrounded.

Watch
Sends "toggle" command, phone wakes in background
MetronomeEngine
Audio starts (background audio mode works), but...
LiveActivityManager
Activity.request() may fail silently in background context
iOS restriction: "Your app must be in the foreground to start a Live Activity"

Workaround: Open the phone app once (creates the Live Activity), then use the watch as remote. Updates to existing activities work from background.

Future fix: ActivityKit push notifications (iOS 17.2+) can start Live Activities from server-side. See GitHub Issue #6.

Limitation Phone app force-quit by user

▶

If the user swipes up from the app switcher to force-quit the phone app, WCSession cannot wake it. This is an iOS restriction.

Watch
Sends command via sendMessage — fails (no app to receive)
Watch
Falls back to transferUserInfo (queued), but won't deliver until app is manually reopened

Mitigation: The watch UI shows "Phone not connected" when isReachable is false. Audio stops on the phone because the process is gone.

Handled Widget intent runs in separate process

▶

When a Live Activity button is tapped, the Intent runs in the widget extension process. It cannot directly access MetronomeEngine.

Widget Extension Process
ToggleMetronomeIntent reads/writes SharedMetronomeState (UserDefaults app group)
Widget Extension Process
Posts Darwin notification (commandStart/commandStop) to notify app process
App Process
StateChangeObserver picks up Darwin notification, drives MetronomeEngine

Why Darwin notifications? NSNotificationCenter is process-local. Darwin notifications (CFNotificationCenter) cross process boundaries. UserDefaults app group provides shared data storage.

Engine State Machine

MetronomeEngine has two primary state variables: isPlaying: Bool and bpm: Int (range 150-230). Every state transition updates 5 systems: SharedMetronomeState, LiveActivityManager, MPNowPlayingInfoCenter, PhoneSessionManager (watch), and the audio timer.

Stopped (isPlaying = false)

Timer: nil (no ticks)
PlayerNode: stopped
Now Playing: "180 SPM" / "Stopped" / .paused
Live Activity: shows START button
Watch: shows play icon (green)

Transitions out:

togglePlayback() → Playing
handlePlayCommand() → Playing
incrementBPM() / decrementBPM() → Stopped (new BPM)
setBPM() → Stopped (new BPM)

Playing (isPlaying = true)

Timer: fires every 60/BPM seconds
PlayerNode: scheduling tick buffer on each fire
Now Playing: "180 SPM" / "Playing" / .playing
Live Activity: shows STOP button
Watch: shows stop icon (red)

Transitions out:

togglePlayback() → Stopped
handleStopCommand() → Stopped
handleInterruptionBegan() → Stopped (saves wasPlaying)
incrementBPM() / decrementBPM() → Playing (debounce 0.3s)
